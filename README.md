# STEP

## 概要

本プロジェクトは、製品プロモーション特設サイトの構築提案に対する回答リポジトリです。
**「低予算・短期間・保守工数ゼロ」**というRFPの至上命題に基づき、Jamstack構成（microCMS × Cloudflare Pages）を採用。非エンジニアの広報担当者が、エンジニアのサポートなしで安全・迅速に情報を発信できる仕組みを構築しました。

---

## 技術スタック

- HTML / SCSS / JavaScript
- Gulp（ビルドツール）
- microCMS（ヘッドレスCMS）
- Cloudflare Pages（想定ホスティング）

---

## 技術選定の理由

### microCMS

- 非エンジニアでも直感的に更新可能
- APIベースでシンプルな構成
- サーバー管理不要で低コスト運用

### HTML / SCSS / JavaScript

- 短期間での制作を考慮し、一番慣れているスキルを活用。
- 不要な複雑性を排除し、シンプルな構成を優先。

### Gulp

- SCSSコンパイル、圧縮、リロードなどの開発効率を改善
- 既存の設定を使って素早く開発を始められる

### Cloudflare Pages

- 無料で利用可能
- CDNによる高速配信
- サーバーレスで保守不要

---

## 技術選定の比較と判断根拠

本プロジェクトの制約（低予算・短期間・保守工数極小）に基づき、以下の比較検討を行いました。

| 候補技術 | 採否 | 不採用の理由（判断の根拠） |
| :--- | :---: | :--- |
| **WordPress** | ❌ | サーバー保守（パッチ当て）やセキュリティ対策の継続的なコストが発生し、RFPの「保守負担の最小化」に反するため。 |
| **Next.js / Nuxt** | ❌ | 高機能だが学習・構築コストが高く、今回の小規模な特設サイトにおいてはオーバーエンジニアリングと判断。 |
| **Google Sheets** | ❌ | 柔軟だがCMSとしての管理画面の使い勝手（画像管理等）が、非エンジニアの運用には不向きであるため。 |
| **HTML/JS + microCMS** | ✅ | **採用。** 既存スキルを最大活用し短期間での完遂を保証。サーバーレス構成により保守コストをゼロにできるため。 |

## CMSデータ設計（microCMS）

### API名

`news`

### フィールド構成

| フィールド | 型 | 必須 | 表示名 |
|-----------|----|------|------|
| title_ja | テキストフィールド | ◯ | 日本語タイトル |
| title_en | テキストフィールド | - | 英語タイトル |
| body_ja | テキストエリア | ◯ | 日本語本文 |
| body_en | テキストエリア | - | 英語本文 |
| thumbnail | 画像 | - | サムネイル |

### 設計意図

シングルエンドポイントによる多言語管理
本プロジェクトでは、1つのコンテンツ内に言語ごとのフィールドを保持する**「1コンテンツ・多言語フィールド方式」**を採用しました。

この設計を選択した理由は以下の通りです：

1. **データ整合性の確保**: 日英の記事が1つのIDで紐付くため、言語切り替え時に「記事が迷子になる（404エラー）」リスクを完全に排除しました。
2. **運用の効率化**: 管理画面上で日英の入力を同画面で確認でき、翻訳漏れや更新ミスを物理的に防止します。
3. **APIリクエストの最適化**: 1回のAPIコールで全言語のデータを取得できるため、フロントエンドでの言語切り替え処理（JS）を高速化させています。

## 多言語対応

- `/ja/`（日本語）と `/en/`（英語）のディレクトリ構造を採用し、言語ごとに独立した静的HTMLを提供。
- 言語切り替えボタン押下時に、現在の記事IDを保持したまま別言語の対応ページへ遷移させるロジックを実装し、ユーザー体験（UX）を向上させました。

---

## SEO対応

多言語サイトにおける検索エンジン最適化（SEO）を考慮し、以下の技術的施策を講じています。

### 1. 言語ごとのURL正規化
- ディレクトリ単位で言語を完全に分離（`/ja/` および `/en/`）し、検索エンジンが各言語のコンテンツを独立してインデックスできるように構成しました。
- 各ページに `hreflang` 属性（`rel="alternate"`）を実装し、日本語版と英語版の適切な関連付けを明記しています。

### 2. JavaScriptによる動的なメタタグ・構造化データ制御
ニュース詳細ページにおいて、microCMSから取得したデータに基づき、以下の項目をJavaScriptで動的に書き換えています。
- **タイトル・ディスクリプション**: 記事内容に応じた適切なメタタグの生成。
- **OGP / Twitter Card**: SNSシェア時に最適な画像と説明文が表示されるよう絶対パスで指定。
- **Canonical属性**: 重複コンテンツを避けるため、正規URLを動的にセット。

### 3. 構造化データ（JSON-LD）の実装
- **ニュース詳細ページにおいて**、Schema.orgの `NewsArticle` スキーマをJSON-LD形式で動的に生成。
- 記事のタイトル、画像、公開日、著者情報を構造化して伝えることで、Google検索結果でのリッチリザルト表示を促進し、クリック率（CTR）の向上を図っています。

### 4. 多言語フォールバック設計
- 英語版の記事が未作成の場合でも、ユーザーを迷わせないよう日本語版への誘導リンクを表示する等、UXとクローラビリティを両立させた設計を行いました。

## ディレクトリ構成

```
.
├── src/                # 開発ソースファイル
│   ├── ja/             # 日本語版コンテンツ（HTML）
│   ├── en/             # 英語版コンテンツ（HTML）
│   ├── js/             # JavaScript (ES Modules形式)
│   ├── scss/           # スタイルシート (Sass/SCSS)
│   └── images/         # 画像アセット
└── dist/               # 本番配信用ファイル（Gulpビルド成果物）
    ├── ja/
    ├── en/
    └── assets/         # コンパイル・最適化済みのJS/CSS/画像
```
- src：開発用
- dist：本番配信用

Cloudflare Pagesでは dist をデプロイ対象としています。

## 運用フロー

コンテンツの更新から公開まで、エンジニアの手を介さない自動配信フローを構築しています

1. **コンテンツ入稿**: 運営者が `microCMS` の管理画面で記事を作成・公開。
2. **Webhook連携**: 公開完了後、microCMS から `Cloudflare Pages` へ Webhook 通知を送信。
3. **自動ビルド**: Cloudflare Pages が通知を受け取り、最新の API データを取得して `dist/` フォルダを再生成（ビルド）。
4. **デプロイ**: 世界中のエッジサーバーへ最新の静的ファイルが自動配信され、サイトに即時反映。

## セキュリティ

### APIキーの管理方針
本プロジェクトでは、`main.js` 内にAPIキーを直接記述する構成を採用しています。

**判断の根拠:**
1. **読み取り専用権限の限定:** 使用しているmicroCMSのAPIキーは「コンテンツ取得（GET）」のみに権限を絞っており、データの改ざんや削除のリスクを排除しています。
2. **公開情報の取り扱い:** 対象コンテンツが一般公開される「ニュース記事」であり、秘匿性の高い個人情報等を含まないため、クライアントサイドでの保持を許容範囲と判断しました。
3. **開発スピードの優先:** 短期間の試験導入において、環境変数設定によるオーバーヘッドを避けるためのミニマムな構成です。

### 本番運用を見据えたセキュリティ拡張案
実運用（商用環境）への移行時には、以下の構成をとることでさらなる安全性を担保します。
- **リファラ制限:** microCMS管理画面にて、特定のドメインからのリクエストのみを許可する設定の導入。
- **APIキーの隠蔽:** Cloudflare Workers等のサーバーレス関数をプロキシとして介在させ、フロントエンドからAPIキーを完全に排除する構成へのアップグレード。

## 今後の拡張案（ロードマップ）

本提案の構成をベースに、サイトの成長や運用要件の変化に応じて以下の拡張が可能です。

### 1. ユーザー体験（UX）の向上
- **404専用ページの作成**: 存在しないURLへアクセスした際、サイトのデザインを維持した404.htmlを表示し、トップページへのスムーズな回遊を促します。
- **レスポンシブ対応の最適化**: 多様なデバイスサイズに合わせ、より緻密なブレイクポイント設定と画像最適化（`srcset`等）を検討します。
- **JS/CSSのモジュール化**: 
    - JavaScript: ファイル分割（ES Modules）を徹底し、関心の分離によるメンテナンス性を向上。
    - SCSS: 変数・Mixin・Extendをより高度に活用し、ブランドカラーや共通パーツの変更を容易にする設計へ強化。

### 2. コンテンツ・運用性の強化
- **リッチエディタの導入**: 現在のテキストエリアからリッチエディタに移行し、記事本文への画像埋め込みやリンク設置、文字装飾を可能にします。
- **カテゴリー・タグ機能**: ニュース件数が増加した際、閲覧者が目的の情報を探しやすくするためのフィルタリング機能を実装。

### 3. SEO・パフォーマンスの極大化
- **独自ドメインの適用**: `Cloudflare Registrar`等を利用し、ブランディングを強化。
- **静的サイトジェネレーター（SSG）への移行**: 
    - 段階的な移行として `EJS` などのテンプレートエンジンの導入を検討。
    - 大規模化の際は、`Astro`, `Nuxt`, `Next.js` などのモダンフレームワークへ移行し、ビルド時間の短縮とさらなる表示高速化を実現。

### 4. 保守・運用の高度化
- **自動テストの導入**: ビルド時にリンク切れチェックやHTMLバリデーションを自動実行し、品質管理を自動化。
